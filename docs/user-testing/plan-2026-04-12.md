## Hypothèses à valider

Ces 4 hypothèses ont été formalisées par l'Agent 11 (codex batch audit 2026-04-11) mais n'ont **pas encore été validées par test utilisateur réel**. Elles restent ouvertes jusqu'au premier cycle modéré.

### A11-01 — Checkout mobile friction (P1 hypothèse)

Le checkout concentre mode invité, récupération implicite du compte, création d'adresse inline, CGV et confirmation Stripe dans **une seule page**.

**Hypothèse à valider :** sur iPhone/Android, des testeurs guest et client hésitent sur :
- le statut "invité vs connecté"
- la prochaine étape
- le niveau de confiance avant paiement

**Réf code :** `apps/web/src/app/(shop)/checkout/page.tsx`

**À mesurer :** taux de complétion, temps de tâche, hésitations observées, verbalisations sur la confiance.

### A11-02 — Repair / Urgence compréhension (P1 hypothèse)

Les parcours `/reparation` et `/urgence` promettent diagnostic, devis, suivi et parfois créneau rapide, mais la **compréhension réelle** de ces promesses reste à mesurer.

**Hypothèse à valider :** des testeurs clients confondent ce qui est :
- immédiat
- estimatif
- confirmé

Et **savent mal comment récupérer** après erreur ou absence de créneau.

**Réf code :** `apps/web/src/app/(shop)/reparation/page.tsx`, `apps/web/src/app/(shop)/urgence/page.tsx`

**À mesurer :** clarté des délais annoncés, compréhension du workflow devis → acceptation → réparation, recovery après erreur.

### A11-03 — Backoffice commandes + SAV (P2 hypothèse)

Les workflows admin commandes et SAV sont riches mais denses, avec filtres, transitions de statut, notes et actions secondaires.

**Hypothèse à valider :** admin, technicien et support réussissent la tâche, mais avec des écarts notables de :
- temps
- erreurs évitables
- hésitation selon leur rôle

**Réf code :** `apps/web/src/app/(admin)/admin/commandes/page.tsx`, `apps/web/src/app/(admin)/admin/sav/page.tsx`

**À mesurer :** temps moyen par tâche, error rate, différences de pattern par rôle.

### A11-04 — Mon compte compréhension (P2 hypothèse)

`/mon-compte` mélange login, inscription, récupération de commandes/SAV, export et suppression de compte.

**Hypothèse à valider :** clients existants comprennent mal où retrouver leur suivi et les nouveaux utilisateurs hésitent sur le bon point d'entrée, surtout après un échec de connexion.

**Réf code :** `apps/web/src/app/(shop)/mon-compte/page.tsx`

**À mesurer :** time-to-information (combien de temps pour retrouver un suivi SAV), taux de succès login sur mobile, abandon après erreur.

## Scope proposé

Parcours critiques seulement, **sans audit large du repo** :

- `apps/web/src/app/(shop)/checkout/page.tsx`
- `apps/web/src/app/(shop)/reparation/page.tsx`
- `apps/web/src/app/(shop)/urgence/page.tsx`
- `apps/web/src/app/(shop)/mon-compte/page.tsx`
- `apps/web/src/app/(admin)/admin/commandes/page.tsx`
- `apps/web/src/app/(admin)/admin/sav/page.tsx`

## Personas

| Persona | Objectif | Priorité |
|---|---|---|
| **Guest** | Premier achat, mobile prioritaire | Haute |
| **Client** | Retrouver commande ou suivi SAV | Haute |
| **Admin** | Traiter une commande et mettre à jour son statut | Moyenne |
| **Technicien** | Faire avancer un ticket SAV | Moyenne |
| **Support** | Retrouver un ticket, expliquer l'état et lancer la bonne action | Moyenne |

## Matrice devices

- **Mobile iPhone Safari** (priorité 1 — notre cible principale démo)
- **Mobile Android Chrome** (priorité 2)
- **Desktop Chrome ou Safari** (priorité 3 — validation que le desktop n'est pas cassé)

## Tâches critiques

### Guest
- Acheter un produit jusqu'au paiement

### Client connecté
- Se connecter
- Retrouver une commande
- Retrouver un suivi SAV
- Déposer une demande `/reparation`
- Déposer une demande `/urgence` avec ou sans créneau

### Admin
- Retrouver une commande
- Changer le statut
- Ajouter un tracking number

### Technicien / Support
- Ouvrir un ticket SAV
- Faire une transition de statut
- Comprendre quand relancer

## Mesures obligatoires

- **Task success / abandon** (binaire)
- **Temps de tâche** (de l'intention à la complétion)
- **Points de confusion verbalisés** (notes observateur)
- **Capacité de recovery** après erreur ou impasse (binaire)
- **Clarté perçue de confiance** : paiement, devis, délai, suivi, consentement (échelle 1-5)

## Taille minimale du cycle

- **1 testeur par persona** (minimum viable)
- **Chaque parcours public** vu au moins une fois sur mobile et une fois sur desktop
- **Session modérée** de 45-60 min par testeur
- **Enregistrement** (avec consentement explicite) pour review a posteriori

## Actions recommandées

### Cycle 1 (semaine 16-17)
1. Lancer un premier cycle court de **5 sessions modérées** (une par persona)
2. Prioriser : checkout mobile, `/urgence`, `/reparation`, récupération depuis `/mon-compte`
3. **Séparer les scénarios admin, technicien et support** au lieu d'un unique test "backoffice"

### Cycle 2 (après fixes)
4. Reclasser `A11-01` à `A11-04` après sessions réelles en :
   - **Finding confirmé** (si l'hypothèse tient) → remonter dans le registre central comme P1/P2 code finding
   - **Non-finding** (si l'hypothèse est invalidée) → marquer Accepted / Closed
   - **Angle à rouvrir** (si de nouveaux problèmes apparaissent) → créer de nouveaux findings A11-0X

### Cycle 3+ (post go-live)
5. Cycle récurrent trimestriel post go-live commercial
6. Mesurer les A/B d'amélioration UX livrées entre chaque cycle
